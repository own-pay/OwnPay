DOMAIN A — SECURITY VULNERABILITIES
--------------------------------------------------------------------------------

For every topic below, apply each check to what you discovered during planning.
These are investigative questions, not assumptions. If a feature does not exist
in this codebase, mark it not applicable and explain why.

A1  Authentication and Session Management

    Find the session and authentication code by reading the actual files.
    Then answer the following:

    - How are sessions created, stored, and destroyed?
    - How are API keys and tokens generated? Is the entropy source
      cryptographically secure, meaning random_bytes or random_int?
    - Is hash_equals used for credential comparison to prevent timing attacks?
    - What happens after repeated failed login attempts? Is there lockout or delay?
    - Can a session be fixed before authentication and reused afterward?
    - Are remember-me tokens stored hashed, with an expiry, and single-use?
    - If JWTs are used, is the algorithm validated server-side? Is alg:none
      accepted? Is the signing secret strong and not hardcoded?
    - Are passwords hashed with bcrypt or argon2 at an adequate cost factor?

A2  Authorization and Access Control

    - Is authorization enforced at the application logic layer for every
      operation, or only at the routing layer where it can be bypassed?
    - For every resource that carries an ID such as a transaction, invoice,
      customer record, payout, or webhook, is ownership verified in the query
      or handler before returning data?
    - Is any merchant ID or user ID accepted from the request body or query
      string and trusted without verifying it matches the authenticated session?
    - Can a lower-privilege role reach higher-privilege operations by altering
      a parameter, header, or URL?

A3  Injection

    - Find every location where database queries are constructed. Are all
      values parameterized? Is there any dynamic table name, column name,
      or ORDER BY clause built from user input?
    - Find every location where shell commands are executed. Does any part of
      the command derive from user-supplied or externally sourced data?
    - Find every dynamic include or require. Can the included path be
      influenced by user input?
    - Is unserialize called anywhere on data from cookies, sessions, the
      database, cache, or the request body?

A4  Cryptography

    - Find every location where tokens, nonces, IDs, or secrets are generated.
      What random function is used? Is it cryptographically secure?
    - Is sensitive data encrypted at rest? If so, what algorithm and key
      management approach is in use?
    - Are webhook signatures verified? What algorithm is used? Is the
      comparison performed in constant time?
    - Is md5 or sha1 used for any security purpose?
    - Are any hardcoded secret values present in source files?

A5  Cross-Site Scripting

    - Is every piece of user-supplied data HTML-encoded before being rendered
      in a browser context?
    - Is a template engine used? Does it auto-escape output by default?
    - Does any JSON response include user data? Is it encoded safely?
    - Are Content-Security-Policy headers present on HTTP responses?
    - Is stored content from external sources such as merchant names, product
      names, or webhook payloads rendered in HTML without escaping?

A6  Cross-Site Request Forgery

    - Are CSRF tokens present on all state-changing HTML forms?
    - Are state-changing API endpoints protected by CSRF tokens or a scheme
      that provides equivalent protection such as token-based auth with no cookies?
    - What is the SameSite attribute on session cookies?

A7  File Operations

    - Find every file upload handler. How is the file type validated? Can the
      validation be bypassed? Are uploaded files stored inside or outside the
      web root? Are filenames sanitized before storage?
    - Find every file read, download, or include that uses a user-supplied path
      or filename. Is path traversal prevented?
    - Find every unlink call. Is the target path sanitized and confirmed to be
      within an expected directory before deletion?

A8  Server-Side Request Forgery

    - Find every location where the application makes an outbound HTTP request.
      Is the URL or any portion of it derived from user or merchant input?
    - For webhook URL registration, is the destination URL validated against
      an allowed scheme, hostname, and IP range to prevent access to internal
      services or cloud metadata endpoints?

A9  Security Headers and Server Configuration

    - What HTTP security headers are present on responses? Check for
      X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security,
      Referrer-Policy, Content-Security-Policy, and Permissions-Policy.
    - Are sensitive paths such as .env, config, vendor, storage, and .git
      blocked at the web server layer?
    - Is PHP execution disabled inside upload directories?
    - Are verbose error messages and stack traces suppressed outside debug mode?

A10 Dependency Vulnerabilities

    - Review composer.lock against known vulnerability data.
    - Flag any package version with a known CVE relevant to how this
      application uses that package.


DOMAIN B — PAYMENT AND BUSINESS LOGIC
--------------------------------------------------------------------------------

B1  Transaction State Integrity

    - Map the complete payment state machine from the actual code. Confirm
      that every state transition is enforced server-side and cannot be skipped.
    - Can a payment be captured without a valid prior authorization?
    - Can the same payment be captured twice via concurrent requests? Are
      database-level row locks or atomic conditional updates in use?

B2  Amount and Currency Integrity

    - Is the payment amount set only by the server at initiation, or can the
      client or payer modify it between steps?
    - Is financial arithmetic performed using integers representing minor
      currency units, or is floating-point arithmetic used? Floating-point
      math on monetary values is a defect.
    - Can an amount of zero or a negative value be submitted and processed
      successfully?
    - Is the currency code validated server-side against an allowed list?

B3  Webhook Authenticity and Idempotency

    - Is the incoming webhook signature verified before any processing occurs?
    - Is there a deduplication mechanism to prevent the same webhook event
      from being processed more than once, such as storing a unique event ID?
    - What happens if webhook processing fails partway through? Is state
      left consistent and is the retry path safe?
    - Can an external party forge a webhook request to trigger a payment
      success on a pending transaction?

B4  Refund Logic

    - Can a refund amount exceed the original captured amount?
    - Can a refund be issued against a transaction that is not in a
      refundable state?
    - Can the same transaction be refunded twice through concurrent requests?
      Is the cumulative refunded amount tracked and enforced atomically?

B5  Payout Logic

    - Is the merchant available balance checked atomically before a payout
      is triggered? Can a race condition produce a negative balance?
    - Can the same payout be triggered and processed twice?
    - Are payout destination details protected from unauthorized modification?
    - Is there a manual approval step for payouts? Can it be bypassed through
      a direct API call?

B6  Multi-Tenant Data Isolation

    - For every database query that retrieves or modifies a financial record,
      is a tenant or merchant scope filter present and impossible to remove
      through user input?
    - Can Merchant A access or trigger actions on Merchant B data through
      any code path?

B7  Merchant Verification

    - Can an unverified or suspended merchant process live payments?
    - Can a merchant self-elevate their verification status by including a
      parameter in an API request?
    - Are KYC documents stored outside the web root and behind access controls?

B8  Fee and Commission Integrity

    - Is the platform fee calculated exclusively on the server side?
    - Can a merchant or payer supply parameters that reduce or eliminate the
      platform fee?

B9  Rate Limiting and Abuse Surface

    - Is payment initiation rate-limited per payer IP and per merchant?
    - Are authentication endpoints protected against brute force?
    - Are webhook registration and API key creation rate-limited per account?

B10 Audit Trail

    - Is every financial event written to a log? This includes payment
      initiation, authorization, capture, settlement, refund, chargeback,
      payout, configuration change, and admin action.
    - Is the log append-only? Can any record be deleted or modified?
    - Are all admin actions attributed to the acting user in the log?


DOMAIN C — CODE QUALITY AND ENGINEERING
--------------------------------------------------------------------------------

C1  PHP 8.2 Correctness

    - Dynamic properties used without the AllowDynamicProperties attribute,
      which was removed in PHP 8.2
    - Deprecated dollar-brace string interpolation syntax
    - Loose equality comparisons in security-critical or state-checking contexts
      where strict comparison is required
    - Type juggling risks where coercions on IDs or status codes may produce
      unexpected truthy or falsy results
    - Use of rand or mt_rand for any security-sensitive token or identifier
      generation instead of random_bytes or random_int

C2  Error and Exception Handling

    - Are exceptions caught at every entry point and handled gracefully?
    - Is the at-sign error suppression operator used anywhere?
    - Do catch blocks silently swallow exceptions without logging them?
    - Can an unhandled exception expose a stack trace in the HTTP response?

C3  Input Validation

    - Is there a centralized input validation layer, or is validation ad hoc?
    - Are there inputs with no length limit that could enable large-payload
      denial of service attacks?
    - Are numeric inputs validated before use in financial calculations?

C4  Database Layer

    - Are multi-step financial operations wrapped in database transactions
      to ensure atomicity?
    - Are there N+1 query patterns that could amplify server load?
    - Are foreign key and frequently-filtered columns indexed?
    - Can soft-deleted records appear in queries that should exclude them?

C5  Secrets and Configuration

    - Are secrets loaded exclusively from environment variables or a secrets
      manager, never from committed source files?
    - Are any real credential values present in the codebase, including in
      example or test files?
    - Is there a clear and enforced separation between development and
      production configuration?

C6  Logging Hygiene

    - Are logs structured for machine parsing?
    - Is any PII, card data, or secret value written to a log?
    - Can user input be used to forge or corrupt log lines?

C7  Dead Code and Debug Artifacts

    - Are debug output calls such as var_dump, print_r, die, or dd present
      outside of test files?
    - Are there TODO, FIXME, or HACK comments indicating disabled or
      incomplete security controls?
    - Are test routes or admin bypass routes present in the production codebase?

C8  Architecture Soundness

    - Is business logic confined to service or domain classes rather than
      mixed into controllers and request handlers?
    - Are payment gateway integrations abstracted behind interfaces that
      enforce consistent input validation and output normalization?
    - Is the routing layer auditable and visible in one place?


DOMAIN D — INFRASTRUCTURE AND DEPLOYMENT
--------------------------------------------------------------------------------

D1  Web Server Configuration

    Find any nginx, apache, or .htaccess configuration files and check:
    - Are application internals such as config, vendor, storage, .env,
      and .git blocked from web access?
    - Is PHP execution disabled in upload directories?
    - Are server version tokens suppressed from response headers?

D2  Container Configuration

    If Docker or Compose files are present, check:
    - Is the application process running as a non-root user?
    - Are secrets injected via environment or mounted volumes rather than
      baked into the image?
    - Are image versions pinned to specific digests rather than mutable tags?

D3  Documentation and Hardening Guidance

    - Does the setup documentation instruct the operator to change all
      default credentials before going live?
    - Is there a post-installation security checklist for self-hosters?
    - Is there a security.txt file and a responsible disclosure policy?


DOMAIN E — COMPLIANCE
--------------------------------------------------------------------------------

E1  PCI-DSS Awareness

    - Does the codebase ever store raw card numbers, CVV values, or track data?
      If yes, this is a critical finding regardless of any other context.
    - Is the cardholder data flow documented so that self-hosting operators
      understand their own PCI compliance scope?

E2  Data Privacy

    - Is personally identifiable information stored with a defined retention
      period and a deletion mechanism available?
    - Are IP addresses anonymized or hashed in logs?

E3  AGPL-3.0 Compliance

    - Are all composer dependency licenses compatible with AGPL-3.0?
    - Does the codebase include license headers in source files?
    - Is the network-use source disclosure requirement documented for operators
      who deploy Own Pay as a hosted service?


SECTION 4 — FINDING FORMAT
================================================================================

Use this exact structure for every issue found. No exceptions.

    FINDING-[NNN]
    -----------------------------------------------------------------------
    Severity     : CRITICAL | HIGH | MEDIUM | LOW | INFO
    Domain       : A-Security | B-Business Logic | C-Code | D-Infra | E-Compliance
    Category     : e.g. Race Condition, IDOR, Missing Auth, Type Juggling
    Location     : exact/file/path.php line number or function name
                   List all affected locations if multiple files are involved
    -----------------------------------------------------------------------
    Description  : What the flaw is and what assumption it violates.
                   Be precise. One focused paragraph.

    Evidence     : The exact source code lines that prove the flaw exists.
                   No paraphrasing. Paste the actual code.

    Attack Path  : Step-by-step description of how an attacker triggers this.
                   Use specific HTTP requests or code call sequences.

    Impact       : The concrete consequence — financial loss, account takeover,
                   data exposure, or system compromise. Quantify where possible.

    Fix          : Corrected code, configuration change, or architectural change
                   required. Not generic advice. Actual working code.

    Reference    : CWE identifier and OWASP ASVS section where applicable
    -----------------------------------------------------------------------

Severity Definitions:

    CRITICAL   Direct financial loss, unauthenticated access to financial
               operations, payment fraud at scale, full account takeover,
               unauthenticated remote code execution, or exposure of all
               user data.

    HIGH       Authenticated privilege escalation, cross-merchant data access,
               webhook forgery enabling false payment confirmation, stored XSS
               in a privileged panel, double-spend via race condition, or SSRF
               reaching internal network services.

    MEDIUM     Information leakage, CSRF on sensitive operations, missing rate
               limits on abuse-prone endpoints, insecure defaults, floating-point
               financial arithmetic, or unvalidated redirects.

    LOW        Best-practice gaps, minor configuration hardening, informational
               leakage with limited practical exploitability.

    INFO       Observations, improvement suggestions, compliance notes.
