QUEST 2: HIGH-VOLUME CONCURRENCY, REDIRECT GAPS & PAYMENT INTEGRITY
--------------------------------------------------------------------------------
Audit the core payment transaction services, refund/dispute processors, double-entry
bookkeeping components, and redirect handlers. Inquire and verify:

1. 100k Simultaneous Hits on Payment API:
   - Analyze exactly what happens if 100,000 users hit the payment API at the exact
     same millisecond.
   - Will the system experience lock contention, latency, or total connection pool exhaustion?
   - Will it fail gracefully for some users, or will it crash?
   - Trace the database isolation levels and transaction locks (`FOR UPDATE` / row-level locks).
     Is there ANY risk of race conditions, deadlocks, or cross-user data mismatches
     (e.g., User A's transaction matching User B's record, or balances getting corrupted)?
2. The Checkout Redirect Divergence Gaps:
   - **Scenario A (Slow IPN / Session Desync)**: A user's payment is 100% successful in the
     external gateway, but when they redirect back to the checkout page, they see "Payment Failed"
     or "Pending" instead of "Success". Trace how this desync occurs (e.g., gateway webhook / 
     Instant Payment Notification (IPN) delayed or blocked by the network, lack of active 
     polling on the frontend, or expired session variables).
   - **Scenario B (Status Tampering / Callback Failure)**: The checkout page displays "Payment Successful",
     but after the customer is redirected back to the merchant's external website, the merchant's
     system marks the transaction as "Failed". Audit the return callback token verification. Is 
     it vulnerable to status parameter tampering? Does it fail to verify signature hashes, or 
     is there an async callback mismatch between the merchant and gateway?
3. Double-Entry Accounting Balancing:
   - Does every financial movement record entries across Ledger Accounts, Ledger 
     Transactions, and Ledger Entries tables?
   - Is there a strict Journal Balance Constraint? Verify that the sum of debits
     MUST equal the sum of credits for every transaction, rolling back on imbalance.
4. GAAP Compliance & Directionality:
   - Are account balances adjusted strictly according to GAAP rules? 
   - Verify that DR increases and CR decreases Asset/Expense accounts, and CR increases 
     and DR decreases Liability/Equity/Revenue accounts.
5. Scoped Cloning Integrity:
   - When a repository or service uses tenant scoping, does it return a cloned 
     instance? 
   - Verify that the codebase captures and utilizes this cloned returned instance, 
     rather than referencing the original unscoped/stale instance.
6. Refund Calculations:
   - Are refund fees and net amounts calculated proportionally and atomically, 
     preventing refunds from exceeding the original transaction amount or double-refunding?

QUEST 4: THE SMS INGESTION, PAIRING & PARSING ENGINE
--------------------------------------------------------------------------------
OwnPay utilizes companion mobile devices to ingest and match carrier receipt SMS 
messages to automate offline payment confirmations. Find and audit the SMS services,
parsers, device pairing repositories, and cron verification jobs. Inquire and verify:
1. Parameter Mismatch Bug Hunt (CRITICAL):
   - Locate where the SMS parsing engine is called by the MFS/payment matching service
     (e.g., when processing incoming SMS data). 
   - Verify the argument order! Check if there is a severe parameter mismatch where the 
     sender is passed as the message body, or the body is passed as the sender. If this
     exists, it will completely break all automatic MFS parsing. Trace this call stack!
2. SMS Data Processing & Fail-safes:
   - What happens if the SMS content contains new words, letters, or structures not 
     defined in the regex template? Does the regex engine gracefully fall back to 
     lexical/keyword proximity heuristic analysis, or does it trigger unhandled errors?
3. The Single vs. Multi-Word Reference Problem:
   - Payer references in SMS receipts (like bKash/Nagad) can contain a single word 
     or multiple words with or without spaces (e.g. "School Fee" vs "fee"). 
   - How does the parser capture these? If the regex limits matching to strict 
     alphanumeric patterns (like `[A-Z0-9]`), does a reference with spaces cause the 
     entire regex block to fail to match the transaction? What happens to the payment?
4. Device Pairing & Scoping:
   - When mapping JWT claims, does the pairing service gracefully fallback to the 
     default superadmin ID `1` if the owner column is null, or does it crash?
5. Notification UUID Scoping & Casting:
   - When retrieving device identifiers from API controllers, are UUIDs strictly 
     cast to strings? If cast to integers, does the UUID turn to `0`, bricking all
     device queries?
   - Verify that notification acknowledgements filter strictly by device UUID to 
     prevent cross-brand/cross-device IDOR vulnerabilities.
6. SMS Spoofing & Template Bypass Loophole Audit:
   - Identify if there are any loopholes, bugs, or design errors in the SMS template regex engine, 
     the heuristic fallback engine, or the payment matching job that allow an attacker to bypass 
     real SMS checks with a fake SMS message to verify a transaction.
   - For example, what if the attacker embeds the whitelisted sender name (e.g. "bKash" or "Nagad") 
     inside the SMS body text? How does the parsing engine ensure that the sender identity is verified 
     exclusively from raw carrier headers and paired mobile device metadata, preventing the system 
     from using attacker-controlled body content?
   - What if an attacker attempts to spoof a transaction ID? Are transaction IDs strictly deduped 
     and validated against the database (`op_sms_parsed.trx_id`) globally, or does a collision or 
     replayed message allow an attacker to verify a different user's pending payment link or invoice?
