# OwnPay Core - Attack Surface Map

This map outlines the critical system boundaries and data entry points of the OwnPay architecture, enumerating known attack vectors, required privileges, and the corresponding mitigations verified during the audit.

| Component / Entry Point | Threat Actor Privilege | Attack Vector | Mitigating Control | Status |
| :--- | :--- | :--- | :--- | :--- |
| **Mobile API (Device Pairing)** | Staff (Low Privilege) | **Privilege Escalation**: Exploiting the owner-fallback mechanism during stateless OTP validation to acquire Super Administrator JWT claims. | None. The system blindly falls back to User ID 1. | **CRITICAL FLAW** |
| **Invoice Updates** | Staff / Admin | **Data Destruction / Race Condition**: Submitting an empty items payload or triggering a crash during the line item rebuild loop to zero out the total and orphan records. | None. The update operations are not encapsulated in a database transaction block. | **HIGH FLAW** |
| **Gateway Adapters (Alipay/Easypaisa)** | Unauthenticated | **Authentication Bypass (State Confusion)**: Sending a forged return callback to a sandbox-configured gateway that lacks a hash key. | Mitigated strictly by ensuring the merchant provides keys and operates in `live` mode. Fails open otherwise. | **HIGH FLAW** |
| **Outbound Webhooks** | Admin (Merchant) | **Server-Side Request Forgery (SSRF) & DNS Rebinding**: Registering malicious URLs to scan internal network CIDRs or extract cloud metadata. | `UrlValidator` blocks private blocks and `WebhookDispatcher` pins the resolved public IP to the cURL handler. | **SECURE** |
| **Inbound Webhooks** | Unauthenticated | **Signature Spoofing / Replay**: Fabricating webhook events or replaying stale, legitimate payloads to force arbitrary transaction status updates. | Constant-time `hash_equals()` comparisons and strict `< 300` second signature timestamp skew enforcement. | **SECURE** |
| **MFS / SMS Parsing** | Unauthenticated (Network) | **Encrypted Payload Replay**: Modifying the outer JSON envelope timestamp to bypass `isDuplicate` deduplication on intercepted AES-GCM SMS payloads. | While the parser deduplication can be bypassed, the `SmsVerificationJob` strictly requires `$transaction['status'] === 'pending'`, successfully rejecting double-spend attempts. | **SECURE** |
| **Double-Entry Ledger** | Staff / Admin | **Ledger Imbalance**: Direct API manipulation or race conditions attempting to write debits and credits unequally to system accounts. | `LedgerRepository` enforces transaction wrappers and GAAP-compliant balancing constraints (DR == CR) prior to database insertion. | **SECURE** |
| **Tenant Isolation boundaries** | Staff / Admin | **Cross-Brand Data Leakage (IDOR)**: Modifying sequential primary keys to access or modify resources belonging to another master tenant brand. | Extensive use of the `TenantScope` trait ensures all queries inherently append `merchant_id = ?` to the `WHERE` clauses. | **SECURE** |
