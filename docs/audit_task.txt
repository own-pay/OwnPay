You are conducting the final pre-release audit of OwnPay — a custom PHP
open-source project running a universal plugin architecture. This audit is
the last gate before OwnPay's first public release. Nothing leaves without
your sign-off. Your job is to find everything that is wrong, incomplete,
insecure, inconsistent, or not ready for public consumption.

You have full access to the local machine and the entire OwnPay codebase.
Do not guess. Do not assume. Read the actual code. Follow actual logic.
Trace actual function calls. Everything you report must be grounded in
what the code actually does — not what it appears to do at a glance.

---

## MANDATORY FIRST STEP — UNDERSTAND BEFORE YOU AUDIT

Before running any audit, you must deeply understand OwnPay.
Do not start finding bugs until you understand the system.

Explore and document your understanding of:

  1. The framework architecture — how the core is structured, what the
     layers are, how they relate to each other, what the request lifecycle
     looks like from entry point to response.

  2. The plugin system — how plugins are discovered, loaded, registered,
     and executed. How the hook/filter/event system works. What the plugin
     contract is. What a plugin can and cannot do. How plugins interact
     with core. How plugins interact with each other.

  3. The template engine integration — how Twig is wired into the
     framework. How data flows from the backend to templates. How template
     inheritance and components are structured.

  4. The database layer — how the ORM or query layer works, how schema is
     defined, how migrations are managed, how relationships are modeled.

  5. The authentication and session system — how users are identified,
     authenticated, and authorized across requests.

  6. The frontend architecture — how vanilla JS is organized and loaded,
     how CSS is structured, how assets are managed and versioned.

  7. The complete user flows — trace every path a user can take through
     the system from first contact to every possible end state.

  8. The configuration and environment system — how OwnPay is configured,
     what environment-specific settings exist, how they are loaded.

  9. The error and exception handling system — how errors propagate,
     what is caught, what is logged, what reaches the user.

  10. The public-facing API surface — every interface that the outside
      world (users, developers, plugin authors) interacts with.

Only after you have a confident, documented understanding of all of the
above should you proceed to the audit phases below.

## AUDIT DIMENSION 1 — ARCHITECTURE & STRUCTURAL INTEGRITY

Audit the entire architectural design of OwnPay:

  - Is the core framework structure logically consistent and internally coherent?
  - Are there architectural violations — places where layers reach through
    each other, or where concerns that should be separated are entangled?
  - Are there circular dependencies anywhere in the codebase?
  - Are there dead code paths — classes, methods, files that are never
    reached and serve no function?
  - Is the plugin system boundary clean? Can a plugin accidentally corrupt
    core state? Are there unguarded plugin hooks?
  - Is the request lifecycle complete and correct at every stage?
  - Are there missing abstraction layers that will force breaking changes
    when OwnPay needs to evolve?
  - Is there any architectural assumption baked in that will not scale or
    will break under real-world usage patterns?
  - Does the directory and namespace structure match the architectural intent?
  - Are there any God objects, God classes, or God files doing too much?
  - Is there a consistent and enforced coding standard throughout the core?

---

## AUDIT DIMENSION 2 — PLUGIN SYSTEM COMPLETENESS & CORRECTNESS

The plugin system is OwnPay's core value proposition. Audit it exhaustively:

  - Is the plugin API complete? Can a plugin author reasonably extend every
    part of the system that should be extensible?
  - Are there parts of the system that should be hookable but are not?
  - Is the hook/filter/event contract well-defined and consistent?
    Are priorities handled correctly? Are infinite loops possible?
  - What happens when a plugin throws an exception? Does the entire system
    fail, or is the failure isolated?
  - What happens when two plugins conflict — register the same hook,
    override the same data, or depend on execution order?
  - Is there protection against malicious plugins? (If this is open-source,
    users will install untrusted plugins.)
  - Is plugin discovery deterministic and safe? Can plugin discovery be
    exploited to load arbitrary code?
  - Is there a clear plugin lifecycle (install, activate, deactivate,
    uninstall)? Is each stage implemented completely and safely?
  - Does the plugin system handle missing dependencies gracefully?
  - Is the plugin API versioned or forward-compatible in any way?
  - Are the contracts a plugin must fulfill clearly enforceable?
    (Interfaces, abstract classes, required methods, etc.)

## AUDIT DIMENSION 3 — SECURITY

Audit the entire codebase for every class of security vulnerability.
Follow every input from its entry point to its final use.
Do not stop at the first use — trace it everywhere it goes.

  INPUT HANDLING & INJECTION
  - Is every user-supplied input sanitized, validated, and escaped
    at every point of use — not just at entry?
  - Are there any SQL injection vectors — raw queries, dynamic table
    or column names, unparameterized inputs anywhere?
  - Are there any XSS vectors — unescaped output in Twig templates,
    unescaped JS variable injection, HTML attribute injection?
  - Are there any command injection vectors — exec, shell_exec, system,
    passthru, proc_open, or similar with user-controlled input?
  - Are there any path traversal vectors — file reads, includes,
    or requires with user-controlled paths?
  - Are there any LDAP, XML, or template injection vectors?
  - Does Twig autoescaping cover every output surface? Are there any
    raw/safe filter usages that are not justified?

  AUTHENTICATION & AUTHORIZATION
  - Is the authentication system complete and correct?
  - Are there any authentication bypass vectors?
  - Is authorization enforced at every action — not just at the route
    level but also at the data level?
  - Are there any Insecure Direct Object Reference (IDOR) vulnerabilities —
    places where a user can access or modify another user's data by
    changing an ID?
  - Are privilege escalation paths possible?
  - Are there any unauthenticated endpoints that should require auth?
  - Is session management secure? (Session fixation, hijacking,
    proper invalidation on logout, secure/httpOnly cookie flags)
  - Are CSRF protections in place on every state-changing request?
  - Are there any race condition vulnerabilities in auth flows?

  CRYPTOGRAPHY & SECRETS
  - Are any secrets, keys, or credentials hardcoded anywhere?
  - Are passwords hashed with a modern, appropriate algorithm?
  - Are any weak or deprecated cryptographic functions used?
  - Is any sensitive data stored or transmitted without adequate protection?
  - Is randomness generated securely wherever it matters?

  FILE OPERATIONS
  - Are there any arbitrary file upload, read, write, or delete
    vulnerabilities?
  - Are uploaded file types validated correctly — not just by extension
    or MIME type but by content inspection where appropriate?
  - Are file paths constructed safely?

  CONFIGURATION & DEPLOYMENT SECURITY
  - Are there any sensitive defaults that should be changed before release?
  - Are there any debug modes, verbose error outputs, or development
    flags that could be left enabled in production?
  - Are error messages user-facing in a way that leaks internal
    implementation details, paths, or query structures?
  - Are security headers set appropriately?
  - Is there any sensitive information exposed in public-facing directories?

## AUDIT DIMENSION 4 — DATABASE SCHEMA & DATA LAYER

  - Is the schema normalized to an appropriate level for the use case?
  - Are there missing indexes on columns used in WHERE, JOIN, or ORDER BY?
  - Are there missing foreign key constraints where relationships exist?
  - Are there missing NOT NULL constraints where nulls should be impossible?
  - Are data types appropriate — not oversized, not undersized, not
    ambiguous (e.g., storing numbers as strings)?
  - Is there a migration system? Is it complete, reversible, and safe
    to run on an existing database?
  - Are there any N+1 query patterns — loops that execute queries
    instead of loading data in bulk?
  - Are transactions used where atomicity is required?
  - Are there any schema inconsistencies — tables or columns that are
    defined but never used, or used but never defined?
  - Is soft delete implemented consistently wherever it should be?
  - Is there any data that should be encrypted at rest but is not?
  - Are there any missing unique constraints that would allow data
    duplication that should be impossible?

## AUDIT DIMENSION 5 — BUSINESS LOGIC COMPLETENESS

Follow every business flow from start to finish. Find every gap.

  - Map every business process the system is intended to support.
  - For each process, trace the complete happy path from beginning to end.
    Confirm it works completely.
  - For each process, identify every edge case and error case.
    Confirm each is handled explicitly — not left to fall through.
  - Find every place where a business rule is partially implemented —
    where the code does step 1 and step 3 but silently omits step 2.
  - Find every place where a business rule is assumed but not enforced —
    where the system trusts that the caller has already validated something
    it never actually validates.
  - Find every place where the same business rule is implemented
    differently in different parts of the codebase — creating inconsistency.
  - Find every incomplete feature — functionality that is scaffolded,
    stubbed, or partially built but not complete enough for release.
  - Find every TODO, FIXME, HACK, or equivalent comment in the codebase.
    Evaluate whether the underlying issue is a release blocker.
  - Find every place where error cases silently fail — where a failure
    produces no error, no log, and no user feedback.
  - Find every place where the system can reach an undefined or
    inconsistent state that there is no recovery path from.

## AUDIT DIMENSION 6 — USER FLOW & EXPERIENCE COMPLETENESS

  - Map every user-facing flow in the system.
  - Trace every flow from entry to completion. Identify any broken steps,
    missing steps, or dead ends.
  - Identify any flow that can leave a user stranded — no success path,
    no error message, no way to proceed or go back.
  - Identify any flow that produces inconsistent outcomes depending on
    timing, browser, or external state.
  - Are all form validations present, correct, and consistent
    between frontend and backend?
  - Are all error messages user-intelligible — not raw exception text,
    not missing, not misleading?
  - Are success states complete — does every successful action
    confirm itself clearly to the user?
  - Are loading and pending states handled — does the UI indicate
    when an operation is in progress?
  - Are there any flows where the frontend and backend have drifted
    out of sync — where the UI expects a contract the backend no
    longer fulfills?

## AUDIT DIMENSION 7 — CODE QUALITY & STANDARDS COMPLIANCE

  - Run PHPStan at level 9 across the entire codebase.
    Every reported error is a finding. Categorize by severity.
  - Are there any type inconsistencies that PHPStan would catch
    but that also represent real runtime risk?
  - Is there consistent and correct use of nullable types, union types,
    and return type declarations throughout?
  - Are there any functions or methods that do too many things
    and should be decomposed?
  - Is error handling consistent — are exceptions used appropriately,
    are error codes used consistently, are there mixed patterns
    that create confusion?
  - Is there dead code — unreachable branches, unused variables,
    imported namespaces never used, methods never called?
  - Are there any infinite loop risks, unguarded recursion risks,
    or memory exhaustion risks?
  - Is logging present and appropriate — are significant events logged,
    are errors logged with enough context to debug, are sensitive
    values not logged?
  - Is the test coverage — if any exists — accurate, meaningful,
    and testing the right things?

## AUDIT DIMENSION 8 — FRONTEND (VANILLA JS & CSS)

  - Are there any JavaScript errors or uncaught exceptions in any
    page or interaction flow?
  - Are there any console.log, console.warn, or console.error calls
    left in production frontend code?
  - Are there any XSS vectors introduced by the JavaScript layer —
    DOM manipulation with unescaped data, innerHTML with dynamic content?
  - Is there any sensitive data — tokens, user PII, internal paths —
    being stored in localStorage, sessionStorage, or cookies without
    appropriate protection?
  - Are all AJAX and fetch requests to the backend using correct
    authentication, CSRF protection, and error handling?
  - Are all API contracts between frontend and backend aligned —
    are requests and responses shaped as the backend actually returns?
  - Is the CSS consistent — are there conflicting rules, unused
    declarations, or visual states that break under certain conditions?
  - Are there any frontend flows that depend on timing or race
    conditions between async operations?
  - Is the frontend accessible? Are there critical missing ARIA labels,
    broken keyboard navigation, or contrast failures?

## AUDIT DIMENSION 9 — OPEN SOURCE RELEASE READINESS

OwnPay is being released publicly. Audit for open-source release standards:

  - Is there a complete and accurate README that explains what OwnPay is,
    how to install it, and how to get started?
  - Is there a LICENSE file that is correct and complete?
  - Is there a CONTRIBUTING guide?
  - Is there a CHANGELOG?
  - Is there a SECURITY policy (how to responsibly disclose vulnerabilities)?
  - Are all development dependencies, debug configurations, and test
    fixtures excluded from the distribution?
  - Are there any hardcoded values, local paths, private API keys,
    internal URLs, or personal references in the codebase that must
    be removed before public release?
  - Are environment-specific configurations externalized and documented?
  - Is there a clear and correct installation process that works on
    a clean environment?
  - Is the plugin developer documentation sufficient for a third-party
    author to build a plugin without reading core code?
  - Are all public-facing interfaces (classes, methods, hooks, filters)
    documented with correct docblocks?
  - Is the version scheme defined and applied consistently?

## AUDIT DIMENSION 10 — CROSS-CUTTING CONCERNS

  - Internationalization and localization — is text hardcoded or
    properly externalized? Are date, time, and number formats handled
    for different locales?
  - Concurrency — are there any race conditions possible under
    concurrent requests? Are shared resources protected?
  - Performance — are there any obviously expensive operations that
    run on every request without caching? Any blocking operations
    that should be async?
  - Dependency management — are all third-party dependencies pinned
    to known good versions? Are any dependencies abandoned, vulnerable,
    or incompatible with the stated PHP version requirements?
  - PHP version compatibility — is the codebase consistent with the
    PHP version range OwnPay claims to support? Are there any functions
    or features used that are unavailable or deprecated in supported versions?
  - Upgrade path — if OwnPay is already installed somewhere and a new
    version is released, is there a safe upgrade path?

## HOW TO CONDUCT EVERY FINDING

For every issue you discover, do not stop at the symptom.

  1. Read the actual code causing the issue.
  2. Understand why it is an issue — the real root cause, not the surface
     manifestation.
  3. Trace where else in the codebase the same issue exists or the same
     pattern is repeated.
  4. Determine the impact — what can go wrong, for whom, under what
     conditions.
  5. Determine whether it is a release blocker or can be deferred.

Do not mark anything as "possibly" or "might be" unless you have genuinely
exhausted your ability to determine the truth from the codebase. If you
cannot determine something from static analysis, say so explicitly
and state what would be required to confirm.
