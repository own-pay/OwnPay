# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

OwnPay is a self-hosted payment automation and Universal Fintech CMS & SaaS Platform (AGPL-3.0, v0.1.0 pre-release). PHP 8.2+, MySQL/MariaDB, Tailwind CSS + Flowbite UI. Clean-slate architecture with a universal plugin system — all extensions (gateways, themes, feature plugins) implement one `PluginInterface`.

## Architecture

### Layers

```
index.php                  # Router entry point (?page=<route>)

.htaccess                  # Apache: front-controller + deny lists for sensitive paths
web.config                 # IIS equivalent of .htaccess
nginx.conf.example         # Nginx deployment template
src/                       # SOA Layer (OwnPay\ namespace, PSR-4)
  Bootstrap.php            # Initializes DB singleton, autoloader
  Core/Database.php        # PDO singleton (fetchAll, fetchOne, transactional)
  Http/                    # Router, Request, RequestContext, JsonResponse
  Middleware/              # 9 middleware classes (Session, CSRF, RateLimiter, etc.)
  Controller/              # Admin controllers + Frontend/ + PluginController
  Service/                 # 20+ business logic services
  Repository/              # 12 repositories with BaseRepository + TenantScope trait
  Security/                # FieldEncryptor, PiiMasker, LogSanitizer
  Event/EventManager.php   # Pure OOP singleton hook/event engine (sole hook API)
  Plugin/                  # Universal plugin system:
    PluginInterface.php    #   ONE contract for ALL extensions (gateway/theme/plugin)
    PluginManifest.php     #   Immutable manifest.json parser
    PluginLoader.php       #   Boot orchestrator (runs on every request)
    PluginRegistry.php     #   In-memory registry + JSON cache
    PluginInstaller.php    #   8-step ZIP install pipeline with security scanning
    PluginSandbox.php      #   Token-based code scanner + capability enforcer
    PluginMigrator.php     #   Transactional DB migration runner for plugins
    Capability.php         #   Backed enum (10 capabilities)
  Cron/CronJobRunner.php   # Cron runner with advisory locking + plugin job registration
app/                       # Procedural Layer
  core/functions.php       # 78 helper functions (getData, insertData, money_*, etc.)
  core/adapter.php         # Session/auth bootstrap, middleware, PluginLoader::boot()
  admin/                   # Admin dashboard views (HTML + PHP)
  install/                 # Setup wizard: index.php + master_install.sql + .htaccess
                           #   .installed marker locks installer after success (WordPress-style)
  modules/gateways/        # Payment gateway plugins (implement PluginInterface)
  modules/themes/          # Checkout themes (implement PluginInterface)
  modules/plugins/         # Feature plugins (implement PluginInterface)
```

### Request Flow

1. `index.php` receives `?page=<route>`, sets security headers (CSP, HSTS)
2. `adapter.php` initializes session, CSRF, permissions via middleware classes
3. Route dispatches to Controller (`::handle($data, $ctx)`)
4. Controller uses Services for business logic, Repositories for persistence
5. Legacy controllers use `getData()`/`insertData()`/`updateData()` helpers

### Plugin System (EventManager + PluginInterface)

ALL extensions (gateways, themes, feature plugins) implement ONE universal `PluginInterface`:
```php
interface PluginInterface {
    public function register(EventManager $events): void;  // Hook registration
    public function boot(): void;                           // Post-registration
    public function activate(): void;                       // One-time setup
    public function deactivate(): void;                     // Cleanup
    public function uninstall(): void;                      // Destructive teardown
    public function info(): array;                          // Metadata
    public function fields(): array;                        // Settings fields
}
```

**EventManager** is the sole hook/event API — no procedural wrappers:
```php
$em = EventManager::getInstance();
$em->addAction('payment.transaction.completed', [$this, 'handler'], owner: 'my-plugin');
$em->addFilter('checkout.gateways', [$this, 'filter'], owner: 'my-plugin');
$em->doAction('system.boot');
$value = $em->applyFilters('invoice.total', $amount);
$em->removeAllByOwner('my-plugin');  // Bulk remove on deactivation
```

**Plugin structure:** Each plugin lives in `app/modules/{gateways|plugins|themes}/<slug>/` with a `manifest.json` and entrypoint PHP file. See `docs/architecture/Universal_Extensibility_Plan.md` for full specification.

**Boot sequence** in `adapter.php`: Middleware -> `PluginLoader::boot()` -> Route dispatch. PluginLoader reads from JSON cache (`storage/plugins/cache/plugin_registry.json`) or DB fallback, then calls `register()` on each, `boot()` on each, and fires `system.boot`.

### Key Globals

- `$db_prefix` - Table prefix (default: `op_`)
- `$global_response_brand` - Current brand context
- `$global_response_user` - Current user session data
- `$admin_type` - `admin` or `staff`
- `$permissions` - User permission array

## Database Conventions

- **Table prefix:** `op_` (configurable via `DB_PREFIX` env var)
- **Modern data access:** `OwnPay\Service\CrudService::select/insert/update/delete/optimisticUpdate/count` — returns arrays directly. Use this for all new code.
- **Legacy helpers:** `getData()`, `insertData()`, `updateData()`, `deleteData()` in `app/core/functions.php`. `getData()` returns a JSON *string* — always `json_decode()` before use. Being phased out in favor of `CrudService`.
- **NULL handling:** DB NULLs flow through as `null` in PHP. Legacy `"--"` sentinel is being phased out. Check with `empty()` or `=== null`, never `=== '--'`.
- **Parameterized queries:** Always use named params (`:param`). Never interpolate user input into SQL.
- **updateData() signature:** `updateData($tableName, $columns, $values, $condition, $whereParams = [])`
- **Money math:** Always use `money_add()`, `money_sub()`, `money_mul()`, `money_div()`, `money_round()` (BC Math). Never use `+`, `-`, `*`, `/` for monetary values.

## Code Conventions

### Controller Pattern (Admin)

```php
namespace OwnPay\Controller;
use OwnPay\Http\RequestContext;

class ExampleController {
    public static function handle(array $data, ?RequestContext $ctx = null): void {
        global $db_prefix, $global_response_brand, $admin_type, $permissions;
        $ctx = $ctx ?? RequestContext::fromGlobals();
        // ... action handling via $data['action'] ?? ''
    }
}
```

### Input Sanitization

- `sanitize_html($value)` - HTML entity encoding + strip tags (for display)
- `clean_input($value)` - Trim only (for DB storage)
- `escape_string($value)` - DEPRECATED, alias for `sanitize_html()`

### Logging

```php
safe_log("PII-safe message: user $userId");           // PII auto-redacted
\OwnPay\Service\Logger::app()->info("message");    // Structured logging
\OwnPay\Service\Logger::security()->warning("x");  // Security channel
```

### CSS/UI

- Framework: Tailwind CSS 3.4 + Flowbite components
- Source: `assets/css/source.css` -> Output: `assets/css/admin.css`
- CSP uses nonce-based script-src: `$csp_nonce` variable in index.php
- Admin pages in `app/admin/dashboard/` use `load_content(title, url, navId)` SPA pattern
- Frontend JS uses `opFetch(action, data, opts)` for AJAX with auto CSRF rotation

### Gateway / Theme / Feature Plugins

All plugins in `app/modules/{gateways|plugins|themes}/<slug>/`. Must have:
- `manifest.json` — Plugin metadata, capabilities, hooks, cron, migrations, admin_menu
- Entrypoint PHP file (e.g. `Gateway.php`, `Plugin.php`, `Theme.php`) implementing `PluginInterface`
- Gateway-specific behavior (capture, webhook) wired via EventManager hooks
- Admin UI for plugins at `?page=plugins` (list, install ZIP, activate, deactivate, delete, settings)

### Permission System

```php
hasPermission($permissions, 'module_name', 'view|create|edit|delete', $adminType)
```


## File Organization Rules

- New business logic goes in `src/Service/`
- New data access goes in `src/Repository/` (extend `BaseRepository`, use `TenantScope` trait)
- Admin HTML views stay in `app/admin/dashboard/`
- All plugins (gateways, themes, features) in `app/modules/{gateways|plugins|themes}/<slug>/` with `manifest.json`
- Plugin infrastructure in `src/Plugin/` namespace
- Hook/event system in `src/Event/EventManager.php` (singleton, sole hook API)
- Never put HTML generation in `src/Controller/` classes
- Tests in `tests/Unit/` following `Tests\Unit\<namespace>` convention
- All new PHP files: `declare(strict_types=1)`, PSR-4 namespacing

## Security Checklist

- All cookies use `SameSite=Strict`, `Secure`, `HttpOnly`
- CSRF tokens rotated per-request via `CsrfMiddleware`
- Rate limiting on login (5/min per IP) and API (per-key sliding window)
- Webhook signatures use HMAC-SHA256 with `hash_equals()`
- CSP header: nonce-based `script-src`, `style-src 'unsafe-inline'` (required by Tailwind CDN)
- Session IDs regenerated on login via `session_regenerate_id(true)`

## Schema & Migrations

- **Fresh installs** use `app/install/master_install.sql` (v2.1, 36 tables, `op_` prefix, partitioning + JSON-schema validation included). Loaded by the installer wizard.
- **Legacy migrations** in `migrations/` (`004`, `005`, `006`) apply only to upgrades from pre-v0.1.0 deployments; greenfield installs do NOT need them.
- Modify `master_install.sql` for any new core tables; never split into separate files post-v2.1.

## Common Pitfalls

- `getData()` returns a JSON *string*. Always `json_decode()` before use.
- `insertData()` auto-discovers columns via `SHOW COLUMNS` - only pass columns that exist.
- The `get_env()` function auto-creates missing env rows. It returns `''` for unset values, not `null`.
- Gateway `class.php` files are loaded dynamically via `require_once`. Class name must match slug pattern: `BkashMerchantGateway` for slug `bkash-merchant`.
- `$data` in controller `handle()` comes from `$_POST` (AJAX). Use `$data['action']` for routing.
- **Installer is locked post-install** by `app/install/.installed` marker + `op-config.php` existence + DB probe. To re-run installer: delete the marker AND `op-config.php` (recommended: just delete `app/install/` entirely).
- **`OWNPAY_INIT` constant** must be defined by the bootstrap chain (`index.php` → `adapter.php`) before any `app/`, `src/`, or `errors/` file loads. Direct file access returns 403.
- **All brand-prefix identifiers use `op_`/`OP_`** (not `ap_`/`AP_`). The codebase was rebranded from AnirbanPay → OwnPay. Anything still containing `ap_` or `AP_` is a regression — see `docs/name_change/installer_security_audit.md`.
